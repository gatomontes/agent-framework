# Citadel Runtime Sovereignty

## Purpose

This document defines the constitutional relationship between The Citadel and execution runtimes.

A runtime is not automatically trusted because it is powerful.

Every runtime operates under Citadel law.

---

# Runtime Definition

A runtime is any execution environment capable of:

- invoking cognition
- loading skills or personas
- performing actions
- producing artifacts
- interacting with tools
- delegating work
- emitting operational state

Examples include:

- OpenClaw
- Codex-oriented systems
- Open-Agent
- local orchestration shells
- future autonomous execution layers

---

# Sovereignty Principle

The Citadel governs.

Runtimes execute.

A runtime may implement operations differently.

It may not redefine doctrine unilaterally.

---

# Runtime Admission Requirements

A runtime becomes Citadel-compliant only if it can:

1. Accept execution contracts.
2. Emit operational status.
3. Preserve artifact references.
4. Surface uncertainty.
5. Support escalation.
6. Respect verification requirements.
7. Preserve consequence tiers.
8. Participate in restoration flow.
9. Avoid silent continuation after failure.

A runtime that cannot satisfy these requirements may still operate experimentally.

It must not be treated as constitutionally trusted.

---

# Runtime Classes

## Experimental

Useful for exploration.

Not trusted for important or critical work.

## Assisted

Can execute contracts and report status but still depends heavily on human supervision.

## Operational

Supports verification, escalation, restoration, and coherent artifact handling.

## Sovereign-Compliant

Fully respects Citadel doctrine and can participate in long-running operational ecosystems.

---

# Runtime Boundaries

Runtimes may:

- execute
- report
- request clarification
- delegate when permitted
- preserve artifacts
- emit operational state

Runtimes may not:

- self-authorize trust
- suppress verification failures
- bypass review gates
- alter consequence tiers silently
- redefine final disposition semantics
- conceal uncertainty

---

# Runtime Failure Semantics

When a runtime fails, it must emit explicit operational state.

Failure examples:

- execution failure
- verification unavailable
- context exhaustion
- recursion instability
- authority conflict
- restoration required
- runtime degradation

Silent failure is doctrinal failure.

---

# Runtime Portability

The Citadel intentionally separates:

- governance
- execution
- cognition
- orchestration
- verification

This allows:

- runtime replacement
- cognition replacement
- orchestration redesign
- independent verification systems
- institutional continuity despite runtime evolution

---

# Constitutional Rule

A runtime is infrastructure.

Trust belongs to verified operational flow, not to the runtime itself.
