# Cognitive Footprint Governance

## Status

Canonical Citadel doctrine.

This doctrine defines how Citadel bounds the cognitive burden created by memory, personas, planning structure, logging, and reusable capability surfaces.

---

# Purpose

Citadel requires cognitive-footprint governance because institutional systems decay when continuity surfaces accumulate faster than they can be interpreted, verified, or governed.

Without footprint discipline:
- memory becomes undifferentiated residue
- persona proliferation creates semantic overlap
- planning decomposes into theatrical topology
- logs overwhelm operational meaning
- activation surfaces multiply beyond mission value
- operators lose reconstructability through sheer volume

Footprint governance exists to preserve institutional readability under continuity pressure.

---

# Core Principle

Accumulation is not continuity.

More memory, more personas, more plans, and more logs do not automatically produce more intelligence.

Citadel must preserve:
- bounded cognition
- readable coordination
- interpretable memory
- justifiable activation surfaces
- survivable operational traceability

---

# Cognitive Footprint Classes

Citadel recognizes at least five major footprint sources:

## 1. Memory Footprint

Includes:
- operational memory artifacts
- continuity notes
- restoration traces
- historical records
- unresolved context shelves

## 2. Persona Footprint

Includes:
- role definitions
- persona overlays
- operator identities
- specialization fragments
- dormant but still-legible capability personas

Citadel should prefer the following progression:

```txt
profession
  -> governed role
    -> activated operator form
      -> persona only when persistent distinction is justified
```

## 3. Planning Footprint

Includes:
- decomposition depth
- topology fanout
- dependency graphs
- sequencing layers
- branching mission structure

## 4. Logging Footprint

Includes:
- runtime logs
- audit traces
- replay records
- event histories
- verification and restoration trace depth

## 5. Capability Footprint

Includes:
- registered capability surfaces
- activation candidates
- bundle-backed utilities
- overlapping operator patterns
- deprecated but still-visible reusable artifacts

---

# Governance Goals

Cognitive-footprint governance exists to preserve:
- interpretability
- selection clarity
- bounded overlap
- operational affordability
- reconstructability
- contradiction visibility
- activation discipline

Footprint minimization is not the goal.

Bounded usable density is the goal.

---

# Footprint Signals

Footprint stress signals include:
- multiple artifacts answering the same operational question
- persona overlap without consequence distinction
- plans whose governance cost exceeds mission value
- logs that cannot be meaningfully replayed by a reviewer
- capability inventories that exceed realistic selection discipline
- stale continuity material surviving without active operational value
- repeated operator confusion over which artifact is canonical

These are survivability warnings.

---

# Memory Footprint Rules

Memory should be curated by operational value, not retained by inertia.

Citadel should distinguish between:
- active continuity material
- historical reconstruction material
- superseded memory
- archival residue

Memory bloat controls may include:
- supersession
- deprecation
- archival
- continuity extraction
- duplicate collapse

Retention without interpretability is memory theater.

---

# Persona Footprint Rules

New personas, role overlays, or operator forms should justify:
- distinct mission consequence
- distinct authority profile
- distinct verification burden
- distinct coordination value

Persona is an expensive operational surface, not a default unit of specialization.

Persona proliferation is prohibited when the new surface merely:
- renames an existing role
- duplicates cognitive burden
- fragments authority unnecessarily
- adds activation ambiguity without mission gain

Not every profession requires a permanent persona.

Where doctrine, topology, and governed role structure are sufficient, personaization should not occur.

Dormant persona inventory should be:
- retired
- merged
- compressed
- or converted into doctrine when its useful attributes are constitutional rather than role-specific

---

# Planning Footprint Rules

Planning structure must remain proportionate to consequence and mission complexity.

Planning bloat signals include:
- decomposition without accountability gain
- topology branches with no distinct authority or verification reason
- sequencing layers that exist only to look sophisticated
- governance fanout that exceeds review capacity

Planning compression may include:
- dependency collapse
- branch reduction
- topology narrowing
- role consolidation
- escalation flattening

Planning reduction must not conceal real dependency or authority structure.

---

# Logging Footprint Rules

Logs are evidence surfaces, not sacred accumulation.

Citadel should preserve enough traceability for:
- reconstruction
- contradiction review
- restoration replay
- audit legitimacy

It should avoid:
- indiscriminate retention
- replay surfaces no reviewer can traverse
- duplicated log channels with no added evidentiary value
- trace granularity that overwhelms the mission tier

Logging density should scale with consequence.

---

# Capability Footprint Rules

Capability surfaces should remain selectable by real operators under real pressure.

A capability inventory becomes noncompliant when:
- overlapping candidates cannot be distinguished meaningfully
- deprecation is absent
- activation choices require archeological effort
- bundle support eclipses the primary capability purpose

Capability inventories should support:
- registration
- review
- suspension
- deprecation
- retirement
- conversion into doctrine where appropriate

---

# Selection Discipline

For any mission, the institution should be able to answer:
- why this memory surface matters now
- why this persona or role remains distinct
- why this planning depth is justified
- why this log density is proportionate
- why this capability surface remains available

If those answers are unavailable, the footprint is already outgrowing governance.

---

# Compression Interface

Cognitive-footprint governance is a practical application layer of:

```txt
/core/doctrine/constitutional-compression.md
```

Compression may be required when footprint stress degrades:
- readability
- selection clarity
- audit survivability
- restoration tractability
- operator cognition

Compression must preserve legitimacy.

---

# Governance Actions

Institutional governance may require:
- persona retirement
- memory consolidation
- planning compression
- log retention narrowing
- capability deprecation
- canonical artifact designation
- footprint review before further growth

Growth without review is drift by accumulation.

---

# Noncompliance

An actor violates cognitive-footprint governance if it:
- multiplies personas without distinct mission value
- preserves memory indiscriminately
- expands planning topology theatrically
- inflates logging density beyond reconstructable value
- retains stale capabilities without governance reason
- hides overlap behind naming variation
- treats accumulation as sophistication

Noncompliance requires review, compression, and when needed, retirement of surplus surfaces.

---

# Final Rule

Citadel must remain cognitively traversable.

If the institution cannot be read, selected, replayed, or governed by its own operators, it has already begun to ceremonialize itself.
