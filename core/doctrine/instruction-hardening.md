# Citadel Instruction Hardening

## Purpose

Instruction Hardening defines how operational instructions, task scrolls, and execution-facing briefs should be shaped before delegated work begins.

It exists to reduce hidden assumptions, fabricated certainty, unsafe irreversibility, and lifecycle corruption.

---

# Core Principle

Ambiguity is not harmless.

Every unresolved decision left inside an instruction becomes an invitation for unauthorized invention.

---

# Hardening Requirements

Before execution, a meaningful instruction should make explicit:

- objective
- scope
- expected output
- constraints
- assumptions
- open questions
- verification requirements
- done condition

If a missing detail materially affects correctness or authority, it must not remain implicit.

---

# Evidence Separation

Instructions should preserve the distinction between:

- confirmed facts
- assumptions
- unresolved questions
- blocked conditions

Inference must not be disguised as observation.

Absence of evidence must not be upgraded into confidence.

When factual claims matter, instructions should make evidence standards explicit before execution begins.

---

# Irreversibility Controls

If an instruction can trigger high-impact or hard-to-reverse action, it must include:

- explicit risk label
- approval boundary
- pre-execution checkpoint
- rollback or containment thinking
- post-action validation step
- stop condition when expected state is not observed

Silent escalation from reversible work into irreversible work is constitutional failure.

---

# State Transition Hygiene

Instructions may not mutate lifecycle state casually.

Before changing state, the operator should know or recover:

- current state
- allowed next states
- forbidden transitions
- required preconditions
- required follow-up records

When transition rules are unclear, read-only recovery or planning is preferable to mutation.

Lifecycle corruption is worse than incomplete progress.

---

# Ownership Boundaries

If work is split across multiple actors, instructions should make explicit:

- owned artifacts
- allowed modifications
- read-only context
- handoff artifacts
- done conditions
- escalation boundaries

Concurrency without ownership boundaries creates integration ambiguity and silent scope drift.

---

# Constitutional Rule

An instruction that depends on guesswork is an undeclared delegation of judgment.
